Como entender os preços

A Anthropic cobra por tokens — a unidade mínima de texto que o modelo processa. Na prática, 1 token equivale a cerca de 4 caracteres em inglês; em português, um pouco mais. Uma página A4 densa tem aproximadamente 600–700 tokens.

Os preços são sempre expressos por milhão de tokens — o que equivale a cerca de 750 mil palavras, ou algo como 1.500 páginas de texto. Na maioria dos usos do dia a dia, você está consumindo frações desse volume.

Há dois tipos de cobrança: input (o que você envia — seu prompt, documentos, histórico da conversa) e output (o que o modelo responde). O output custa entre 3× e 5× mais, dependendo do modelo.

1. Como funciona o consumo de tokens

A Anthropic cobra pelo uso dos modelos Claude com base em tokens. Tokens são as unidades básicas de processamento de texto: cada palavra gera aproximadamente 1 a 2 tokens, dependendo do idioma e da complexidade. O português tende a consumir mais tokens por palavra do que o inglês.

Existem dois tipos de tokens cobrados de forma diferente:

  • Tokens de input (entrada): o texto que você envia ao modelo, incluindo instruções, contexto, documentos anexados e histórico de conversa.
  • Tokens de output (saída): o texto que o modelo gera como resposta. Custam entre 3× e 5× mais que os de input, dependendo do modelo.

O custo total de uma conversa depende de três fatores principais:

  • Qual modelo você usa (Haiku, Sonnet ou Opus)
  • Quanto texto você envia (input) e quanto o modelo gera (output)
  • O comprimento acumulado da conversa: a cada nova mensagem, todo o histórico anterior é reenviado como input
Atenção: conversas longas ficam progressivamente mais caras porque o histórico completo é reprocessado a cada turno. Iniciar conversas novas e focadas é uma das formas mais simples de controlar consumo.

2. O que mudou do Opus 4.6 para o Opus 4.7

O Opus 4.7, lançado em 16 de abril de 2026, é uma atualização direta do Opus 4.6 com o mesmo preço. No entanto, duas mudanças afetam o consumo de tokens e precisam ser compreendidas.

2.1 Tokenizer atualizado

O Opus 4.7 utiliza um novo tokenizer que melhora o processamento de texto, mas faz com que o mesmo conteúdo possa gerar de 1,0× a 1,35× mais tokens de input. Isso significa que enviar exatamente o mesmo prompt pode custar até 35% mais tokens do que antes, dependendo do tipo de conteúdo.

2.2 Mais raciocínio interno (thinking)

O modelo "pensa mais" nos níveis de esforço mais altos, especialmente em conversas longas e em cenários de uso agente. Isso melhora a confiabilidade em problemas difíceis, mas produz mais tokens de output.

2.3 Melhorias concretas

  • Melhor seguimento de instruções: o modelo interpreta orientações de forma mais literal e precisa.
  • Visão aprimorada: aceita imagens com até 2.576 pixels na borda longa (3,75 megapixels), 3× mais que modelos anteriores.
  • Código mais confiável: resultado 13% superior ao 4.6 em benchmarks de coding.
  • Melhor autonomia: ideal para tarefas longas e autônomas, com capacidade de verificar seus próprios outputs.
  • Novo nível de esforço xhigh: controle mais fino entre high e max para balancear qualidade e custo.

2.4 Impacto prático no consumo

Segundo a Anthropic, o efeito líquido é favorável: em suas avaliações internas de coding, o consumo total de tokens melhorou em todos os níveis de effort. No entanto, em uso interativo com muitos turnos, o consumo pode ser maior por causa do raciocínio adicional.

Ponto-chave: quanto mais você fragmenta a tarefa em muitos turnos, mais overhead de tokens você injeta. Com o Opus 4.7, a mudança de mentalidade recomendada é delegar tarefas completas logo no primeiro turno, com contexto estruturado, em vez de fazer micro-ajustes passo a passo.

3. Qual modelo usar para cada tipo de trabalho

A regra prática é simples: comece pelo Sonnet, suba para o Opus só quando o Sonnet falhar de forma consistente, e desça para o Haiku quando for volume, classificação ou texto curto.

3.1 Haiku 4.5 — Rápido e econômico

Custo: US$ 1 / US$ 5 por milhão de tokens. O modelo mais barato e rápido da família.

Quando usar:

  • Classificação de documentos, extração de dados, tagging e triagem
  • Resumos de textos curtos e respostas a perguntas simples (FAQ)
  • Conversões de formato (CSV para JSON, markdown para texto, etc.)
  • Drafts de e-mails curtos e comunicações padronizadas
  • Moderação de conteúdo e pipelines de alto volume
  • Qualquer tarefa onde a alternativa seria "não automatizar porque ficaria caro"

Quando não usar:

  • Raciocínio multi-etapas, análises complexas ou produção de código longo
  • Tarefas com muitas restrições simultâneas ou instruções ambíguas

3.2 Sonnet 4.6 — O modelo padrão para 80–90% do trabalho

Custo: US$ 3 / US$ 15 por milhão de tokens. Melhor relação custo-benefício para tarefas profissionais.

Quando usar:

  • Código do dia a dia: escrever funções, revisar PRs, refatorar módulos
  • Redação: posts, documentação, propostas, relatórios, e-mails complexos
  • Síntese e comparação de múltiplas fontes (pesquisa, análises)
  • Análise de dados tabulares de complexidade média
  • Workflows multi-step com instruções detalhadas
  • Agentes que chamam ferramentas (tool use / function calling)
Desenvolvedores usando Claude Code preferiram Sonnet 4.6 sobre o Opus 4.5 em 59% dos casos. Para a grande maioria do trabalho, Sonnet entrega qualidade equivalente a uma fração do custo.

Quando não usar:

  • Problemas de raciocínio muito difíceis (race conditions, algoritmos novos, 5+ hipóteses simultâneas)
  • Tarefas autônomas de longa duração (horas) onde erros intermediários comprometem tudo

3.3 Opus 4.7 — Para quando a qualidade é inegociável

Custo: US$ 5 / US$ 25 por milhão de tokens. Modelo mais inteligente disponível ao público.

Quando usar:

  • Decisões de arquitetura de software sobre codebases grandes e complexos
  • Debug de problemas difíceis (concorrência, memory leaks, race conditions)
  • Análises jurídicas ou regulatórias com múltiplas restrições interdependentes
  • Modelagem financeira com lógica condicional complexa
  • Agentes autônomos de longa duração que precisam de consistência por horas
  • Tarefas que o Sonnet comprovadamente falhou com consistência
  • Leitura e interpretação de imagens de alta resolução (diagramas, screenshots densos)
Não use Opus "para garantir". Use quando o custo de uma resposta errada justificar o custo do modelo. Se você não consegue explicar por que Sonnet não serve para aquela tarefa específica, provavelmente Sonnet serve.

5. Dicas práticas para reduzir consumo

5.1 Para usuários do claude.ai

  • Selecione o modelo certo na interface. Use o seletor de modelo antes de cada conversa. Não deixe tudo em Opus por padrão.
  • Conversas curtas e focadas. Cada mensagem reprocessa o histórico inteiro. Conversas longas ficam exponencialmente mais caras.
  • Envie apenas o necessário. Não cole PDFs de 100 páginas para perguntar algo específico. Extraia o trecho relevante.
  • Peça concisão. Inclua no prompt: "responda em no máximo X frases" ou "seja breve". Output custa 3–5× mais que input.

5.2 Para desenvolvedores (API / Claude Code)

  • Use o parâmetro effort. No Opus 4.7: low e medium para tarefas simples, high para a maioria, xhigh apenas para problemas realmente difíceis.
  • Prompt caching. Reduz custos de input em até 90% para system prompts e contextos repetitivos. Indispensável.
  • Batch API. 50% de desconto para requisições assíncronas que não precisam de resposta imediata.
  • Delegue tarefas completas no primeiro turno. Com Opus 4.7, micro-turnos geram overhead. Estruture objetivo, escopo, restrições e critérios de aceitação logo no primeiro prompt.
  • Roteamento de modelos. Use Haiku para classificar a requisição, Sonnet para processar, e Opus só para escalar casos difíceis. Pode reduzir custos em 60–70%.
